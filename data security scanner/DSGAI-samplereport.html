Goal

This report scans a software codebase against the OWASP GenAI Data Security Risks and Mitigations 2026 (v1.0) — a framework published by the Open Worldwide Application Security Project in March 2026 that defines 21 data security risks specific to GenAI and agentic systems. The goal is to automatically detect OWASP GenAI data security risks and make it easier for teams to act on them — without requiring every developer or reviewer to manually read and interpret the full OWASP specification. Beyond detection, the automation provides several additional benefits:

  • Shift-left security — catches risks at development time, not at pen test or production incident time
  • Consistent coverage — eliminates human variance in manual reviews; every scan checks all 21 controls every time
  • Live CVE intelligence — automatically cross-references the repo's exact dependency versions against live vulnerability databases, surfacing exploitable CVEs without manual lookups
  • Audit-ready output — generates a shareable, self-contained HTML report that serves as documented evidence of a security review against a published standard
  • CI/CD integration ready — the scan can be run on every pull request or release branch, making GenAI security a continuous practice rather than a one-time checkpoint
  • Prioritised remediation backlog — findings are tiered so teams know what to fix today versus what goes into the architecture backlog

This report is designed to benefit security practitioners, developers, security architects, and engineering managers — see the Who Benefits section below for details on how each audience can use it.

This report contains two sections:
  1. Section 1 — DSGAI Compliance Findings: A full assessment of the codebase against all 21 OWASP DSGAI controls, with per-control status (FAIL / WARN / PASS / N/A), evidence pointers into source files, and prioritised remediation guidance.
  2. Section 2 — CVE Advisory: Real-time CVE data fetched at scan time from the NVD (National Vulnerability Database) and OSV, cross-referenced against the exact dependency versions pinned in this repository. Each CVE is classified as Exploitable, Not Affected, or Unknown and mapped to the relevant DSGAI control.

Executive Summary

The demo-customer-ai Customer Support AI Assistant was assessed against 21 OWASP DSGAI controls. The overall posture is critical — 11 controls fail, with multiple high-severity issues directly exposing live customer PII.

  • The system uses LangChain 0.1.16 with SQLDatabaseChain executing LLM-generated SQL directly against PostgreSQL — no query validator, no read-only enforcement.
  • ChromaDB 0.4.24 is deployed unauthenticated over plain HTTP with ALLOW_RESET=true, exposing all customer knowledge base embeddings.
  • An MCP server with no authentication exposes tools that return customer names, emails, dates of birth, and order history to any network caller.
  • A database password is hardcoded in the LLM system prompt (config.py:8) — the model can leak it in any response.
  • Full prompts and LLM responses are logged verbatim to stdout and telemetry, including all customer PII retrieved from the knowledge base.
  • CVE enrichment is online — 5 CVEs found across 3 DSGAI risk groups: 4 Not Affected, 1 Unknown scope. The LangChain SQL injection pattern (CVE-2023-36189) is not exploitable by version, but the architectural risk (unvalidated LLM-generated SQL) is confirmed present under DSGAI12.
Recommended remediation priority: Remove the password from the system prompt → disable verbose telemetry logging → enforce SQL read-only + validator → add ChromaDB auth and TLS → add MCP authentication → implement tenant isolation in RAG queries.

DSGAI01 FAIL Training Data Privacy BOTHTier 1

Raw documents containing customer PII (email addresses, dates of birth, partial card numbers) are embedded into ChromaDB without any anonymization or PII scrubbing. No presidio, redact, or anonymize patterns found anywhere in the ingestion pipeline.

app/rag/ingestion.py:20 — embedding = self.embeddings.embed_query(doc["content"]) # no PII scan app/rag/ingestion.py:32 — "alice@example.com", "Bob Smith (DOB: 1985-03-12)", "Card ending 4242"
Remediation
  • Add presidio-analyzer + presidio-anonymizer and run on doc["content"] before embedding
  • Remove PII from hardcoded seed documents in ingestion.py:32-34
  • Set allow_training=False / X-Training-Data: false when switching to a real LLM API
DSGAI02 FAIL Agentic Credential Management BUILDTier 1

Database password hardcoded in config.py. The same password is also embedded in the LLM system prompt, making it visible to the model and leakable in responses. K8s secret is base64-only (not encrypted at rest). No vault or secrets manager integration found.

app/config.py:2 — DB_PASSWORD = "Sup3rS3cr3t!" app/config.py:3 — DB_URL = "postgresql://support_user:Sup3rS3cr3t!@postgres:5432/..." app/config.py:8 — SYSTEM_PROMPT contains "our database admin password is Sup3rS3cr3t!" k8s/secret.yaml:11 — base64-encoded password (not encrypted at rest by default)
Remediation
  • Load DB_PASSWORD from environment variable or Vault (os.environ["DB_PASSWORD"])
  • Remove the password from SYSTEM_PROMPT — this is a critical data leak vector
  • Enable Kubernetes EncryptionConfiguration for Secrets at rest, or use External Secrets Operator
  • Scope agent tool grants per session — not all tools to all requests
DSGAI03 WARN Shadow AI & Unauthorized Data Flows BOTHTier 2

No hardcoded external LLM endpoints found (uses FakeListLLM stub). However, LANGCHAIN_TRACING_V2=true is set in both .env.example and docker-compose.yml, sending full trace data (prompts, completions) to LangSmith — an external SaaS — without evidence of a DPA or data residency review.

.env.example:14 — LANGCHAIN_TRACING_V2=true docker-compose.yml:16 — LANGCHAIN_TRACING_V2=true
Remediation
  • Disable LANGCHAIN_TRACING_V2 in production or route to a self-hosted LangSmith instance
  • Document LangSmith as an approved data processor with a signed DPA
  • Add egress policy restricting LLM API calls to approved endpoints only
DSGAI04 FAIL AI Supply Chain Security BUILDTier 1

torch.load() called without weights_only=True — allows arbitrary code execution via pickle deserialization if the model file is tampered. torch and numpy are unpinned (>= range). No model checksum verification. No SBOM generation in CI.

app/models/loader.py:15 — model = torch.load(model_path) # missing weights_only=True requirements.txt:19 — torch>=2.0.0 # unpinned requirements.txt:20 — numpy>=1.24.0 # unpinned
Remediation
  • Change to torch.load(model_path, weights_only=True)
  • Pin torch and numpy to exact versions in requirements.txt
  • Add sha256 checksum verification before torch.load()
  • Add syft SBOM generation step to .github/workflows/ci.yml
DSGAI05 WARN RAG Data Security BUILDTier 2

similarity_search() runs with no access control filter — all documents are returned regardless of sensitivity level. Documents contain customer PII with no access-level metadata. No path traversal protection on the /ingest endpoint's source parameter.

app/rag/retriever.py:22 — results = self.vectorstore.similarity_search(query, k=k) # no filter app/main.py:61 — async def ingest_documents(source: str = "knowledge_base") # unsanitized
Remediation
  • Add filter={"access_level": "internal"} to all similarity_search() calls
  • Tag documents with access_level and sensitivity metadata at ingestion time
  • Validate and allowlist the source parameter on /ingest
DSGAI06 FAIL MCP and Plugin Security BUILDTier 1

MCP server has no authentication — any network client can call all customer data tools. All MCP calls use plain HTTP. No tool argument schema validation. No Kubernetes NetworkPolicy restricting which pods can reach the MCP server.

mcp_server/server.py:4 — no authentication middleware on any route app/config.py:19 — MCP_SERVER_URL = "http://mcp-server:8001" # plain HTTP app/agents/customer_agent.py:62 — httpx.get(f"{MCP_SERVER_URL}/tools/...") # no auth header k8s/deployment.yaml:57 — no NetworkPolicy restricting MCP pod access
Remediation
  • Add bearer token middleware to MCP server: validate Authorization: Bearer <MCP_API_KEY>
  • Pass auth header in customer_agent.py: headers={"Authorization": f"Bearer {MCP_API_KEY}"}
  • Switch to HTTPS or add TLS at the ingress level
  • Add Kubernetes NetworkPolicy to restrict MCP pod access to the API pod only
  • Add Pydantic input validation on all MCP tool arguments
DSGAI07 FAIL Data Lifecycle Management BUILDTier 2

No TTL on session history — AgentExecutor memory grows indefinitely. No right-to-erasure handler. No embedding deletion capability. No data lifecycle constants (RETENTION_DAYS, HISTORY_TTL) anywhere in the codebase.

app/agents/customer_agent.py:44 — no session TTL, no memory eviction policy
Remediation
  • Add ConversationBufferWindowMemory(k=10) or TTL-based memory to AgentExecutor
  • Implement a /gdpr/delete/{session_id} endpoint that purges conversation history
  • Add HISTORY_TTL_DAYS = 30 constant and enforce via a scheduled cleanup job
  • Implement client.delete_collection() or namespace-scoped deletion for ChromaDB

DSGAI11 FAIL Multi-Tenant Data Isolation BUILDTier 1

Single shared ChromaDB collection customer_knowledge_base serves all users with no tenant namespace or filter. Every similarity_search() call is missing a tenant filter — any user's query can retrieve documents belonging to any other user.

app/rag/vectorstore.py:17 — COLLECTION_NAME = "customer_knowledge_base" # shared, no tenant scope app/rag/retriever.py:22 — similarity_search(query, k=k) # no filter={"tenant_id": ...}
Remediation
  • Add filter={"tenant_id": current_tenant} to all similarity_search() calls
  • Tag all documents with tenant_id metadata at ingestion time
  • Consider per-tenant Chroma collections: collection_name=f"{tenant_id}_knowledge"
DSGAI12 FAIL Database Agent Security BUILDTier 1 CVE-2023-36189

SQLDatabaseChain executes LLM-generated SQL directly against PostgreSQL with no query validator, no read-only enforcement, and no DDL protection. The LLM could generate DROP TABLE, DELETE FROM customers, or TRUNCATE orders.

app/agents/sql_agent.py:28 — SQLDatabaseChain.from_llm(llm=self.llm, db=self.db, verbose=True) app/agents/sql_agent.py:43 — result = self.chain.invoke({"query": ...}) # no validation before execution app/config.py:3 — DB connection uses full CRUD account (no read-only user)
Remediation
  • Create a read-only PostgreSQL user and use it for the SQL agent connection
  • Add a SQL validator that rejects any non-SELECT statement before execution
  • Add LIMIT 100 cap enforcement on all agent-issued queries
DSGAI13 FAIL Vector Store Security BUILDTier 1 CVE-2024-3584

ChromaDB initialized with no authentication over plain HTTP. ALLOW_RESET=true in both Docker Compose and Helm — allows wiping the entire vector store without credentials. K8s secret stores values as base64 only.

app/rag/vectorstore.py:12 — chromadb.HttpClient(host=CHROMA_HOST, port=CHROMA_PORT) # no auth docker-compose.yml:45 — ALLOW_RESET=true helm/values.yaml:28 — ALLOW_RESET: "true"
Remediation
  • Add headers={"Authorization": f"Bearer {CHROMA_API_KEY}"} to chromadb.HttpClient()
  • Set ALLOW_RESET=false in all non-test environments
  • Add TLS termination at ingress or configure Chroma with SSL
DSGAI14 FAIL AI Telemetry & Observability Data Security BUILDTier 1

Full prompts and LLM responses logged verbatim including all customer PII. print(response) writes full output to stdout (visible in container logs and any log aggregator). CAPTURE_CONTENT=True and LOG_FULL_RESPONSE=True enabled with no PII redaction.

app/config.py:12-14 — LANGCHAIN_VERBOSE=True, CAPTURE_CONTENT=True, LOG_FULL_RESPONSE=True app/telemetry/logging.py:20 — "prompt": prompt # raw PII logged verbatim app/telemetry/logging.py:34 — print(f"[RESPONSE] session={session_id} response={response}")
Remediation
  • Set CAPTURE_CONTENT=False and LOG_FULL_RESPONSE=False in production config
  • Add presidio PII scrubbing before any logger.info() that includes prompt/response content
  • Remove all print(response) statements from production code paths
  • Log only session_id, token counts, and latency — not content
DSGAI15 FAIL Context Window Data Security BUILDTier 1

System prompt is hardcoded in source and contains the plaintext database password. If the model ever includes system prompt content in its output — a known LLM behaviour — the password is exposed to end users. No context size limits configured.

app/config.py:6-9 — SYSTEM_PROMPT hardcoded with: "our database admin password is Sup3rS3cr3t!"
Remediation
  • Remove the password from the system prompt immediately
  • Load SYSTEM_PROMPT from an environment variable or Vault, not hardcoded in source
  • Add max_context_length truncation to prevent unbounded context accumulation

DSGAI18 WARN Model Output Data Security BUILDTier 2

No output guardrails, moderation layer, or PII detection on LLM responses before returning to the caller. max_tokens not set on LLM calls (currently FakeListLLM stub — real deployment must configure this).

app/main.py:59 — return ChatResponse(answer=response, ...) # no guardrails check
Remediation
  • Add guardrails-ai or llm-guard output validation before returning from /chat
  • Run presidio PII scan on model output and redact before sending to client
  • Set max_tokens=512 (or appropriate limit) on all real LLM calls

DSGAI20 FAIL Inference API Security BOTHTier 1

The /chat endpoint has no authentication and no rate limiting — accepts unlimited requests from any caller. No prompt injection detection layer. MCP tools are likewise unauthenticated.

app/main.py:18-20 — no rate limiting middleware added app/main.py:45 — @app.post("/chat") — no auth dependency, no rate limit decorator
Remediation
  • Add slowapi rate limiter: @limiter.limit("10/minute") on /chat
  • Add API key or OAuth2 Bearer authentication as a FastAPI dependency on /chat and /ingest
  • Add prompt injection detection (e.g. llm-guard input scanner) before passing to AgentExecutor
DSGAI21 WARN Knowledge Store Security BUILDTier 2

The ingestion pipeline writes to ChromaDB without content validation or audit logging. Retrieval path is read-only (good). No version control on KB entries — a malicious document could persistently poison RAG responses.

app/rag/ingestion.py:21-27 — collection.add(documents=[...]) # no content validation app/rag/retriever.py — read-only: similarity_search only, no writes ✓
Remediation
  • Add content validation before collection.add() to reject injection patterns
  • Restrict the /ingest endpoint to admin/service accounts only
  • Add an audit log entry for every document ingested (who, when, source)
