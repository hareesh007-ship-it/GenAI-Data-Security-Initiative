# LLM Top 10 2025 – NIST SP 800-218A

Mapping the [OWASP Top 10 for LLM Applications 2025](https://genai.owasp.org/llm-top-10/)
to [NIST SP 800-218A: Secure Software Development Practices for Generative AI and Dual-Use Foundation Models](https://doi.org/10.6028/NIST.SP.800-218A.ipd)
(Initial Public Draft, March 2024).

NIST SP 800-218A extends the Secure Software Development Framework (SSDF)
with AI-specific practices covering the full lifecycle of generative AI and
foundation model development — from defining training data requirements and
designing secure ML pipelines, through adversarial testing and supply chain
verification, to vulnerability response and root cause analysis. It is the
primary US government reference for secure AI software development and
directly addresses the risks in the OWASP LLM Top 10. Organisations
following SSDF for their conventional software estate can extend that
programme to their AI systems using this mapping. US federal agencies are
directed to align with SP 800-218A under OMB memoranda referencing the SSDF.

---

## SP 800-218A practice groups

| Group | ID | Purpose |
|---|---|---|
| Produce Well-Secured Software | PW | Security requirements, design, reuse, secure coding, review, and testing across the AI development lifecycle |
| Protect the Software | PS | Protecting model weights, training data, pipeline code, and build artefacts from unauthorised access and tampering |
| Respond to Vulnerabilities | RV | Identifying, assessing, remediating, and analysing AI-specific vulnerabilities including emergent behaviours |

### LLM01 – Prompt Injection

**Severity:** Critical

Malicious instructions embedded in user inputs or external content manipulate
LLM behaviour, bypassing safety controls and executing unauthorised actions.
SP 800-218A addresses this through threat modelling of adversarial inputs in
the design phase (PW.2), output and behaviour review (PW.7), adversarial
red-team testing (PW.8), and vulnerability identification procedures for
prompt injection incidents (RV.1).

**Real-world references:**
- EchoLeak (2025) – indirect prompt injection turned Microsoft 365 Copilot
  into a silent exfiltration engine via email content
- Samsung source code leak (2023) – proprietary data surfaced through
  manipulated model queries

#### SP 800-218A mapping

| Practice | ID | Group | Description |
|---|---|---|---|
| Design software to meet security requirements — adversarial input | PW.2.1-PS | PW | Threat model the AI pipeline for adversarial input vectors including direct and indirect prompt injection; document mitigating design decisions |
| Review the software for security vulnerabilities — model behaviour | PW.7.2-PS | PW | Review model outputs and system behaviour for unexpected responses to adversarial inputs; include injection bypass scenarios in code and design reviews |
| Test for security vulnerabilities — adversarial / red-team | PW.8.2-PS | PW | Conduct adversarial testing (red-teaming) against prompt injection vectors before each production release; cover direct, indirect, and multimodal injection paths |
| Identify and confirm vulnerabilities — AI-specific | RV.1.1-PS | RV | Establish procedures to identify prompt injection incidents in production including monitoring, triage, and confirmation workflows |

#### Mitigations

**Foundational**
- PW.2.1-PS: During design, explicitly threat model all external content
  sources that will be processed by the model — emails, documents, web
  content, tool outputs, RAG results — and design structural separations
  between instruction and data contexts
- Establish a policy that all LLM inputs from untrusted sources are treated
  as potentially adversarial; enforce in architecture review gates
- PW.7.2-PS: Include prompt injection bypass scenarios in pre-release
  model behaviour reviews; assign a reviewer responsible for adversarial
  edge cases

**Hardening**
- PW.8.2-PS: Implement a structured red-team testing programme covering
  direct injection, indirect injection via RAG sources, jailbreak patterns,
  and multimodal vectors — gate production releases on red-team sign-off
- RV.1.1-PS: Deploy runtime monitoring for prompt injection indicators;
  define escalation and triage procedures for detected events
- Apply privilege separation — if the model cannot invoke destructive
  tools by design, injection impact is bounded

**Advanced**
- PW.8.2-PS: Extend adversarial testing to cover your specific RAG
  document corpus, tool descriptors, MCP server schemas, and any
  email or calendar processing pipelines
- RV.1.1-PS: Integrate injection detection signals into your SIEM;
  automate session isolation on high-confidence injection indicators
- PW.2.1-PS: Document threat model outputs as a living SDLC artefact;
  refresh on every new tool integration or data source addition

### LLM02 – Insecure Output Handling

**Severity:** High

LLM-generated content passed to downstream components without validation
enables XSS, command injection, SSRF, or SQL injection via AI-generated
content. SP 800-218A addresses this through secure coding requirements for
output handling (PW.5), output security review (PW.7), and model artifact
integrity verification to ensure the output-producing model has not been
tampered with (PS.2).

#### SP 800-218A mapping

| Practice | ID | Group | Description |
|---|---|---|---|
| Secure coding — ML pipeline output handling | PW.5.1-PS | PW | Implement secure coding practices for all code paths that consume LLM output; treat model responses as untrusted input to downstream systems |
| Review for security vulnerabilities — output validation | PW.7.2-PS | PW | Include output injection and unsafe rendering scenarios in pre-release security reviews; verify that output sanitisation controls are present and effective |
| Verify software integrity — model artifact integrity | PS.2.1-PS | PS | Verify that the model artifact serving output has not been tampered with; maintain signed model checksums and verify before deployment |
| Test for security vulnerabilities — output security | PW.8.2-PS | PW | Include output-based injection testing (XSS, SQLi, CMDi via LLM output) in adversarial test suites |

#### Mitigations

**Foundational**
- PW.5.1-PS: Enforce at code review that all code consuming LLM output
  applies context-appropriate sanitisation before rendering or execution
  — HTML encoding for web contexts, parameterised queries for databases,
  shell escaping for command contexts
- Never pass raw LLM output to eval(), exec(), shell commands, or database
  queries — enforce as a mandatory code review checklist item
- PW.7.2-PS: Include output-to-injection scenario reviews in your
  pre-release security gate; reviewer must confirm output handling controls
  for every LLM integration point

**Hardening**
- PW.8.2-PS: Add output injection scenarios to your adversarial test suite
  — test that crafted model responses cannot achieve XSS, SQL injection,
  or command injection in your specific application stack
- PS.2.1-PS: Verify model artifact integrity before every deployment;
  maintain a signed model registry with checksum verification in CI/CD
- Implement output schema validation — define and enforce the structure
  of acceptable model responses per integration point; reject malformed output

**Advanced**
- PW.8.2-PS: Run DAST against all interfaces consuming LLM output in
  your CI/CD pipeline — include in automated release gates
- Deploy a dedicated output security layer (independent of the model)
  that enforces encoding, schema validation, and content policy before
  output reaches any downstream consumer
- PW.5.1-PS: Extend secure coding standards to include LLM output
  handling patterns; train developers on AI-specific injection risks

### LLM03 – Training Data Poisoning

**Severity:** Critical

Attackers corrupt training data or fine-tuning pipelines to embed backdoors
or bias model behaviour in ways baked into the weights and invisible to
standard inference testing. SP 800-218A addresses this as a core training
data protection requirement (PS.1, PS.3), supply chain vetting for datasets
(PW.4), and root cause analysis for poisoning incidents (RV.3).

**Real-world references:**
- Nightshade (2023) – poison pixels successfully corrupted image generation
  model behaviour at scale
- BadNets (academic) – backdoor triggers embedded through poisoned training
  labels, activating only on specific inputs in production

#### SP 800-218A mapping

| Practice | ID | Group | Description |
|---|---|---|---|
| Protect all code from unauthorised access — training data and pipelines | PS.1.1-PS | PS | Protect training data repositories, fine-tuning datasets, and ML pipeline code from unauthorised read, write, and modification |
| Archive and protect software releases — model versioning | PS.3.1-PS | PS | Maintain versioned, integrity-verified training data snapshots and model checkpoints in a secure model registry; enable rollback |
| Reuse existing well-secured software — dataset vetting | PW.4.1-PS | PW | Vet all third-party and external training datasets for provenance, integrity, and potential poisoning before use in any training run |
| Analyse root causes — training data forensics | RV.3.1-PS | RV | When poisoning is detected, conduct training data analysis to identify the poisoned records, their source, and the blast radius |

#### Mitigations

**Foundational**
- PS.1.1-PS: Implement access controls on all training data repositories —
  enforce least privilege; only approved ML engineers can write to training
  datasets, all access logged with full audit trail
- PW.4.1-PS: Establish a training data approval process — all new datasets
  (including third-party and crowd-sourced) must pass provenance review,
  licence review, and statistical quality checks before entering any
  training pipeline
- PS.3.1-PS: Implement training data versioning — every training run must
  reference an immutable, integrity-verified dataset snapshot

**Hardening**
- PW.4.1-PS: Apply anomaly detection to training data before each
  training run — flag statistical outliers, unexpected label distributions,
  and content inconsistent with the source domain
- PS.3.1-PS: Maintain a secure model registry with signed checksums for
  every model version; implement rollback capability to pre-poisoning
  checkpoints on detection
- RV.3.1-PS: Establish a training data forensics playbook — procedures
  for isolating poisoned records, tracing to source, and determining
  influence on model weights

**Advanced**
- Apply differential privacy during training to bound the influence of
  any single training example — measure and track privacy budget
- PW.4.1-PS: Conduct backdoor detection (neural cleanse or equivalent)
  on all new model versions trained on external datasets before promotion
  to production
- RV.3.1-PS: Implement continuous training data monitoring — alert on
  statistical distribution drift that may indicate an ongoing poisoning
  campaign targeting future training runs

### LLM04 – Model DoS

**Severity:** High

Adversarial inputs trigger disproportionate compute, memory, or token
consumption causing denial of service or runaway inference cost. SP 800-218A
addresses this through resource constraint requirements in design (PW.2),
load and adversarial testing for availability (PW.8), and remediation
procedures for availability incidents (RV.2).

#### SP 800-218A mapping

| Practice | ID | Group | Description |
|---|---|---|---|
| Design software — resource and availability constraints | PW.2.1-PS | PW | Include resource limits, rate limiting, and availability requirements as explicit security design requirements for AI inference services |
| Test for security vulnerabilities — load and adversarial testing | PW.8.2-PS | PW | Conduct adversarial load testing and sponge example testing to validate resource consumption limits hold under attack conditions |
| Assess, prioritise, and remediate — availability remediation | RV.2.1-PS | RV | Define remediation procedures for availability incidents including model rollback, rate limit tightening, and cost circuit breakers |
| Review for security vulnerabilities — resource handling | PW.7.2-PS | PW | Include resource exhaustion scenarios in pre-release security reviews; verify that token limits and rate limits are enforced by design |

#### Mitigations

**Foundational**
- PW.2.1-PS: Define token limits, rate limits per user/session/API key,
  and maximum context window size as explicit design requirements —
  document in system design specifications and enforce at architecture
  review
- Implement rate limiting at the application layer before requests reach
  the model — never rely solely on model-level token limits
- PW.7.2-PS: Verify that resource limits are enforced in pre-release
  security review — confirm that limits cannot be bypassed through
  request chaining or parameter manipulation

**Hardening**
- PW.8.2-PS: Include resource exhaustion scenarios in adversarial testing
  — sponge example attacks, token amplification via recursive prompts,
  and context window stuffing
- RV.2.1-PS: Define automated response to consumption anomalies — rate
  limit tightening, session suspension, cost circuit breakers, owner
  notification; test procedures quarterly
- Implement per-tenant and per-session cost budgets with automatic
  suspension on breach — alerting alone is insufficient

**Advanced**
- PW.8.2-PS: Conduct adversarial cost-maximisation testing — identify
  the specific inputs that generate maximum token consumption for your
  model and guard those paths with additional controls
- Implement adaptive rate limiting with real-time system load awareness
  — thresholds adjust dynamically under attack conditions
- RV.2.1-PS: Document recovery time objectives (RTO) and recovery point
  objectives (RPO) for LLM services in your availability remediation plan

### LLM05 – Supply Chain Vulnerabilities

**Severity:** High

LLM applications depend on third-party model weights, datasets, libraries,
and plugins — any of which can be compromised to introduce backdoors or
malicious functionality. SP 800-218A addresses this as a core SSDF concern
extended to AI: vetting third-party AI models and datasets (PW.4), verifying
model artifact and library integrity (PS.2), maintaining a secure model
registry (PS.3), and monitoring for newly discovered supply chain
vulnerabilities (RV.1).

#### SP 800-218A mapping

| Practice | ID | Group | Description |
|---|---|---|---|
| Reuse existing well-secured software — third-party AI model and dataset vetting | PW.4.1-PS | PW | Vet all third-party model weights, datasets, plugins, and libraries before use; verify provenance, licence, and security posture |
| Verify software integrity — model and library supply chain | PS.2.1-PS | PS | Verify integrity of all model artefacts and third-party components using cryptographic signatures and checksums before deployment |
| Archive and protect software releases — model registry | PS.3.1-PS | PS | Maintain a secure, versioned model registry with provenance records for all components; enable auditability and rollback |
| Identify and confirm vulnerabilities — supply chain monitoring | RV.1.1-PS | RV | Monitor for newly disclosed vulnerabilities in third-party AI components; establish a triage process for AI-specific CVEs |

#### Mitigations

**Foundational**
- PW.4.1-PS: Establish an approved sources policy for AI components —
  model weights, datasets, adapters, and libraries must come from
  approved, vetted sources; require sign-off before any new component
  enters a production pipeline
- Maintain a complete ML SBOM for every production AI system — model,
  adapters, fine-tuning datasets, inference libraries
- PS.2.1-PS: Verify cryptographic signatures or checksums for all
  model artefacts before deployment — do not deploy unsigned or
  unverified components

**Hardening**
- PS.3.1-PS: Implement a versioned, access-controlled model registry;
  all promoted model versions must have documented provenance and
  integrity attestation
- RV.1.1-PS: Subscribe to model provider security advisories and
  AI-specific vulnerability feeds; define triage SLAs for supply chain
  disclosures
- PW.4.1-PS: Scan all third-party libraries in the ML stack using
  standard SCA tools — extend your existing dependency scanning
  programme to cover ML dependencies

**Advanced**
- PW.4.1-PS: Conduct backdoor detection on all new model versions from
  third-party providers before production promotion — use neural cleanse
  or equivalent tooling
- PS.2.1-PS: Implement automated supply chain integrity verification
  in CI/CD — block deployment on any component whose signature cannot
  be verified
- Extend ML SBOM to cover runtime dynamic components — MCP servers,
  plugins, and tools fetched at inference time

### LLM06 – Excessive Agency

**Severity:** High

LLMs granted excessive autonomy over tools, APIs, and systems execute
unintended or harmful actions when manipulated. SP 800-218A addresses this
through AI-specific security requirements that define capability constraints
(PW.1), threat modelling of autonomous action paths in design (PW.2), and
model behaviour review for scope violations (PW.7).

#### SP 800-218A mapping

| Practice | ID | Group | Description |
|---|---|---|---|
| Define security requirements — AI capability constraints | PW.1.1-PS | PW | Define explicit security requirements limiting the capabilities, tool access, and autonomous action scope permitted for each AI deployment |
| Design software — autonomy boundary threat modelling | PW.2.1-PS | PW | Threat model all tool integrations and autonomous action paths; design least-privilege tool manifests and human approval gates for irreversible actions |
| Review for security vulnerabilities — autonomy boundary validation | PW.7.2-PS | PW | Review model behaviour for scope violations — verify that the model cannot invoke tools or take actions outside its defined permission manifest |
| Test for security vulnerabilities — adversarial autonomy testing | PW.8.2-PS | PW | Conduct adversarial testing targeting excessive agency through indirect injection and permission escalation scenarios |

#### Mitigations

**Foundational**
- PW.1.1-PS: Document explicit security requirements for each LLM
  deployment that define the maximum permitted tool access, API scope,
  and autonomous action types — treat these as mandatory deployment
  requirements reviewed before go-live
- PW.2.1-PS: Design a tool permission manifest for every LLM integration
  — principle of least privilege applied to each tool; no tool should
  have broader access than the narrowest capability needed
- Require human approval for all irreversible or high-impact actions —
  enforce in architecture, not relying on model judgment

**Hardening**
- PW.7.2-PS: Include scope violation scenarios in pre-release behaviour
  reviews — verify that the model cannot exceed its permission manifest
  even under adversarial input
- PW.8.2-PS: Test autonomy boundaries as part of each release — conduct
  red team exercises targeting excessive agency through indirect prompt
  injection targeting tool invocation
- Log all tool invocations with full parameter capture — feed into
  runtime anomaly detection

**Advanced**
- PW.8.2-PS: Conduct formal adversarial exercises specifically testing
  indirect injection through every data source that feeds the agent;
  permission manifest must hold under worst-case injection
- PW.1.1-PS: Include AI autonomy limits in security design reviews;
  autonomous action scope changes require explicit security sign-off
  and update to the threat model
- Formally specify permitted action graphs — only pre-approved action
  sequences can execute in production

### LLM07 – System Prompt Leakage

**Severity:** High

System prompts containing internal instructions, business logic, or security
controls are extracted by adversaries — enabling targeted attacks against
specific defences. SP 800-218A addresses this through protection of
configuration and pipeline code (PS.1), secure coding practices that
prevent credential embedding in prompts (PW.5), and model output behaviour
review for unintended disclosure (PW.7).

#### SP 800-218A mapping

| Practice | ID | Group | Description |
|---|---|---|---|
| Protect all code from unauthorised access — system prompt confidentiality | PS.1.1-PS | PS | Classify system prompts as sensitive configuration artefacts; apply access controls, version control, and audit logging to all prompt stores |
| Secure coding — no credentials in prompts | PW.5.1-PS | PW | Enforce secure coding standards prohibiting embedding of credentials, API keys, or sensitive configuration data in system prompts or model context |
| Review for security vulnerabilities — prompt disclosure | PW.7.2-PS | PW | Include system prompt extraction scenarios in pre-release security reviews; verify that outputs cannot reveal system prompt content |
| Test for security vulnerabilities — prompt extraction | PW.8.2-PS | PW | Conduct adversarial prompt extraction testing against all known extraction techniques before deployment and after each system prompt change |

#### Mitigations

**Foundational**
- PS.1.1-PS: Classify system prompts as sensitive configuration artefacts
  — version controlled, access controlled (least privilege), change-logged,
  and reviewed on every modification
- PW.5.1-PS: Enforce that no credentials, API keys, PII, or sensitive
  business logic that must remain confidential are embedded in system
  prompts — use environment variables and secrets managers; enforce at
  code review
- PW.7.2-PS: Include prompt disclosure checks in pre-release reviews;
  verify output sanitisation logic is present

**Hardening**
- PW.8.2-PS: Test prompt extraction resistance before each deployment and
  after every system prompt change — cover all known extraction techniques
  applicable to your model family
- Deploy output monitoring to detect response patterns indicative of
  system prompt disclosure — alert and escalate on detection
- Treat system prompt design as a security design review artefact —
  include in SDLC security gates

**Advanced**
- PW.8.2-PS: Conduct structured red team exercises covering all known
  prompt extraction techniques against your specific deployment; document
  residual risk after controls are applied
- Deploy output classifiers trained to detect and block responses
  containing system prompt content
- PS.1.1-PS: Implement secrets detection in CI/CD pipelines that flag
  any prompt configuration file containing potential credentials or
  sensitive patterns before deployment

### LLM08 – Vector and Embedding Weaknesses

**Severity:** Medium

Weaknesses in embedding stores enable adversarial retrieval manipulation,
inference of sensitive information from embeddings, and semantic search
poisoning. SP 800-218A addresses this through protection of embedding stores
and vector databases as sensitive AI artefacts (PS.1), adversarial testing
of retrieval systems (PW.8), and root cause analysis for embedding-related
incidents (RV.3).

#### SP 800-218A mapping

| Practice | ID | Group | Description |
|---|---|---|---|
| Protect all code from unauthorised access — embedding store protection | PS.1.1-PS | PS | Classify embedding stores and vector databases as sensitive AI artefacts; apply access controls, encryption at rest/in transit, and audit logging |
| Test for security vulnerabilities — embedding adversarial testing | PW.8.2-PS | PW | Conduct adversarial testing of retrieval pipelines including embedding inversion, retrieval poisoning, and semantic search manipulation scenarios |
| Analyse root causes — embedding incident forensics | RV.3.1-PS | RV | When an embedding or retrieval security incident occurs, conduct forensic analysis of the vector store, ingestion pipeline, and query patterns |
| Reuse existing well-secured software — vector store vetting | PW.4.1-PS | PW | Vet vector database platforms and embedding model providers for security posture before adoption; review access control capabilities |

#### Mitigations

**Foundational**
- PS.1.1-PS: Classify all embedding stores as sensitive data assets;
  implement read and write access controls, encrypt data at rest and
  in transit, enable comprehensive audit logging
- Apply data quality validation to all content before embedding
  generation — validate source, integrity, and content before ingestion
  into vector stores
- PW.4.1-PS: Vet your vector database platform for security features
  — verify RBAC, encryption, and audit capabilities before deployment

**Hardening**
- PW.8.2-PS: Include embedding manipulation scenarios in adversarial
  testing — test retrieval poisoning, adversarial query responses, and
  embedding inversion scenarios
- Implement anomaly detection on vector store ingestion and query
  patterns — alert on unusual ingestion volumes, unexpected content
  types, or anomalous query distributions
- Rotate embedding models periodically; embeddings from deprecated
  models may have known inversion vulnerabilities

**Advanced**

### LLM09 – Misinformation

**Severity:** Medium

LLMs generate plausible but factually incorrect content that users or
downstream systems act upon. SP 800-218A addresses this through model
output behaviour review for accuracy and factual correctness (PW.7),
adversarial testing covering hallucination and misinformation scenarios
(PW.8), and vulnerability identification procedures for misinformation
incidents in production (RV.1).

#### SP 800-218A mapping

| Practice | ID | Group | Description |
|---|---|---|---|
| Review for security vulnerabilities — output validation | PW.7.2-PS | PW | Include accuracy, hallucination, and misinformation scenarios in pre-release model behaviour reviews; validate output quality against domain benchmarks |
| Test for security vulnerabilities — behavioural and accuracy testing | PW.8.2-PS | PW | Conduct adversarial and behavioural testing covering hallucination rates, factual accuracy, and misinformation generation potential before each release |
| Identify and confirm vulnerabilities — emergent behaviour monitoring | RV.1.1-PS | RV | Establish procedures to detect and triage misinformation incidents in production including user feedback channels, automated fact-checking, and output monitoring |
| Design software — output confidence and citation requirements | PW.2.1-PS | PW | Design AI systems to surface confidence indicators, source citations, and uncertainty signals in output; include as explicit security and quality requirements |

#### Mitigations

**Foundational**
- PW.2.1-PS: Define as an explicit design requirement that AI systems
  in high-stakes domains (medical, legal, financial, safety-critical)
  must surface source citations and uncertainty signals alongside
  generated content
- PW.7.2-PS: Include hallucination and factual accuracy testing in
  pre-release reviews — benchmark against your specific domain before
  each production deployment
- Establish a policy requiring human verification before AI-generated
  content is acted upon in regulated or safety-critical contexts

**Hardening**
- PW.8.2-PS: Build hallucination and accuracy benchmarks for your
  specific deployment domain — measure and track rates per release;
  gate promotion on threshold compliance
- RV.1.1-PS: Implement production misinformation detection — user
  feedback mechanisms, automated fact-checking for verifiable claims,
  output quality metrics fed into risk monitoring
- Deploy RAG grounded on verified, up-to-date sources for domains
  where accuracy is critical; implement retrieval quality validation

**Advanced**
- PW.8.2-PS: Build automated fact-checking pipelines for high-stakes
  output domains — integrate into the inference pipeline with citation
  verification
- RV.1.1-PS: Implement continuous accuracy monitoring with drift
  detection — alert when hallucination rates exceed defined thresholds
  across production traffic
- PW.7.2-PS: Include misinformation risk in security design reviews
  for regulated industries; document residual risk and required
  human oversight procedures

### LLM10 – Unbounded Consumption

**Severity:** Medium

Adversarial inputs trigger disproportionate resource consumption — CPU,
memory, tokens, API costs — causing denial of service or runaway cost.
SP 800-218A addresses this through resource constraint requirements in
design (PW.2), adversarial load and sponge-example testing (PW.8), and
remediation procedures for availability incidents including model rollback
and throttle deployment (RV.2).

#### SP 800-218A mapping

| Practice | ID | Group | Description |
|---|---|---|---|
| Design software — resource and throttling requirements | PW.2.1-PS | PW | Define token limits, rate limits, and cost budgets as explicit security design requirements for all AI inference services; document in design specifications |
| Test for security vulnerabilities — load and adversarial consumption testing | PW.8.2-PS | PW | Conduct adversarial resource consumption testing (sponge examples, token amplification, recursive context injection) before each production release |
| Assess, prioritise, and remediate — availability remediation | RV.2.1-PS | RV | Define and test remediation procedures for availability incidents — rate limit tightening, model rollback, cost circuit breaker activation |
| Verify software integrity — throttle and limit configuration integrity | PS.2.1-PS | PS | Verify that rate limit and resource constraint configurations have not been altered before deployment; treat throttle configuration as a security artefact |

#### Mitigations

**Foundational**
- PW.2.1-PS: Define as explicit design requirements: per-user token limits,
  per-session rate limits, maximum context window, and monthly cost budgets
  — document in system design; review at architecture gate
- Implement rate limiting at the application layer before requests reach
  the model; never rely solely on model-level controls
- PS.2.1-PS: Treat rate limit and throttle configurations as security
  artefacts — version control, access control, and verify before deployment

**Hardening**
- PW.8.2-PS: Include sponge example attacks and token amplification in
  adversarial testing — validate that resource limits hold under worst-case
  adversarial inputs
- RV.2.1-PS: Define and test automated response to consumption anomalies
  — rate limit tightening, session suspension, cost circuit breakers;
  exercise procedures quarterly
- Implement per-tenant cost budgets with automatic suspension on breach;
  alerting alone is insufficient

**Advanced**
- PW.8.2-PS: Conduct adversarial cost-maximisation exercises specific to
  your model deployment — identify maximum-cost input patterns and apply
  additional controls at those paths
- Implement adaptive rate limiting with real-time system load awareness;
  thresholds adjust dynamically under sustained attack
- RV.2.1-PS: Document RTO and RPO for LLM services; include availability
  remediation in your AI incident response plan
